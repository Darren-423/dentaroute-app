# DentaRoute 팀 개발 워크플로우 가이드

## 1. 브랜치 전략

```
Main (프로덕션)
 └── dev (개발/통합)
      └── feature/* (실험적 기능 브랜치 - 필요 시)
```

| 브랜치 | 용도 | 규칙 |
|--------|------|------|
| **Main** | 안정된 프로덕션 코드 | dev에서 이상 없음 확인 후에만 푸시 |
| **dev** | 일일 개발/통합 | 하루 회의 후 여기에 푸시 |
| **feature/*** | 실험적 기능 | dev 아래에서 생성, 검증 후 dev로 머지 |

---

## 2. 일일 개발 사이클

```
1. 하루 회의
   ↓
2. 각자 담당 페이지 수정 작업
   ↓
3. 수정 전 디스코드에 보고 (어떤 파일을 고칠지)
   ↓
4. 해당 페이지만 수정 (다른 사람 담당 페이지 건드리지 않기)
   ↓
5. 수정 완료 후 AI에게 CLAUDE.md 업데이트 요청
   ↓
6. dev 브랜치에 푸시
   ↓
7. 이상 없으면 Main에 푸시
```

## 3. AI(Claude) 행동 규칙

### 3.1 수정 전 반드시 지킬 것

| 규칙 | 설명 |
|------|------|
| **담당 페이지만 수정** | 사용자가 지정한 파일/페이지만 수정한다. 관련 없는 파일은 절대 건드리지 않는다. |
| **수정 전 보고** | 어떤 파일을 어떻게 수정할지 사용자에게 먼저 알려준다. |
| **최소 변경 원칙** | 요청받은 부분만 수정한다. 불필요한 리팩토링, 주석 추가, 코드 정리를 하지 않는다. |

### 3.2 수정 후 반드시 지킬 것

| 규칙 | 설명 |
|------|------|
| **CLAUDE.md 업데이트** | 수정 완료 후 CLAUDE.md에 변경 사항을 반영한다. 다른 팀원의 AI도 이 파일을 읽고 프로젝트 흐름을 파악하기 때문이다. |
| **변경 요약 제공** | 사용자에게 무엇을 변경했는지 간결하게 보고한다. |
| **디스코드 보고용 메시지 제공** | 팀 채널에 공유할 수 있는 변경 사항 요약문을 작성해준다. |

### 3.3 CLAUDE.md 업데이트 방식

CLAUDE.md는 **팀 전체가 공유하는 프로젝트 컨텍스트 파일**이다.
AI가 작업 시작 전 무조건 읽는 파일이므로, 여기에 최신 상태가 반영되어야 한다.

**업데이트해야 하는 경우:**
- 새 파일/화면 추가 시 → 디렉토리 구조, 화면 설명 섹션 업데이트
- 기존 화면 로직 변경 시 → 해당 화면 설명 업데이트
- 데이터 모델 변경 시 → 데이터 레이어 섹션 업데이트
- 새 컴포넌트 추가 시 → 컴포넌트 시스템 섹션 업데이트
- 네비게이션 흐름 변경 시 → 네비게이션 플로우 섹션 업데이트
- 새 API 엔드포인트 추가 시 → 관련 섹션 업데이트

**업데이트 포맷:**
```markdown
## 변경 이력 (최신순)

### [날짜] - 변경 제목
- **수정 파일**: `경로/파일명.tsx`
- **변경 내용**: 구체적인 변경 설명
- **영향 범위**: 어떤 기능/화면에 영향을 주는지
- **작업자**: 이름
```

## 4. 파일 수정 프로토콜

### 4.1 수정 요청 시 AI 동작 순서

```
1. 사용자가 수정할 페이지/기능 지정
   ↓
2. AI가 해당 파일을 먼저 읽고 현재 상태 파악
   ↓
3. 수정 계획을 사용자에게 보고
   (어떤 파일의 어느 부분을 어떻게 바꿀지)
   ↓
4. 사용자 승인 후 수정 진행
   ↓
5. 수정 완료 후:
   a. 변경 사항 요약 제공
   b. CLAUDE.md 업데이트
   c. 디스코드 보고용 메시지 제공
```

### 4.2 수정 금지 사항

- 다른 팀원이 담당하는 페이지 수정 금지
- 사용자가 지정하지 않은 파일 수정 금지
- 공유 파일(`lib/store.ts`, `constants/*` 등)은 반드시 사전 보고 후 수정
- CLAUDE.md 외의 루트 설정 파일(`package.json`, `tsconfig.json` 등) 임의 수정 금지

## 5. 커밋 & 푸시 규칙

### 5.1 커밋 메시지 형식

```
[영역] 변경 내용 요약

- 상세 변경 1
- 상세 변경 2

Co-Authored-By: Claude Opus 4.6 <noreply@anthropic.com>
```

**영역 태그:**
| 태그 | 의미 |
|------|------|
| `[patient]` | 환자 화면 수정 |
| `[doctor]` | 의사 화면 수정 |
| `[auth]` | 인증 관련 수정 |
| `[component]` | 공유 컴포넌트 수정 |
| `[lib]` | 라이브러리/유틸 수정 |
| `[server]` | 백엔드 수정 |
| `[docs]` | 문서 수정 |
| `[fix]` | 버그 수정 |

### 5.2 푸시 순서

```
1. git add (수정한 파일만 명시적으로)
2. git commit
3. git push origin dev
4. 팀 회의에서 확인
5. 이상 없으면 → git push origin main
```

---

## 6. 팀 협업 시 주의사항

| 상황 | 대응 |
|------|------|
| 공유 파일 수정이 필요할 때 | 디스코드에 먼저 보고 → 승인 후 수정 |
| 다른 사람 코드에 버그 발견 | 직접 고치지 말고 디스코드에 보고 |
| 새 파일 생성이 필요할 때 | 사전에 팀에 알리고 CLAUDE.md에 반영 |
| 충돌(conflict) 발생 시 | 임의로 해결하지 말고 해당 파일 담당자와 협의 |

---

## 요약

> **핵심 원칙: "보고 → 수정 → CLAUDE.md 업데이트"**
>
> 모든 수정은 이 3단계를 따른다.
> CLAUDE.md는 팀 전체 AI가 공유하는 프로젝트 브레인이다.
> 여기가 최신 상태여야 다른 팀원의 AI도 올바르게 작동한다.
